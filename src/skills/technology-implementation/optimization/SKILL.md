# Technology Implementation — Optimization Qualification

Loaded when the classifier labels an opportunity `technology-implementation / optimization`.
Your job: confirm the system is already live (this is improvement, not a build), qualify the deal
against the firm's platform alliances and ICP, and compile the package the
**delivery / proposal team** needs to scope a response.

## When this applies
Improving or extending an enterprise system the client already runs: realizing unused capabilities,
upgrading existing software, adding new modules or best-of-breed components, improving third-party
integrations, or extending existing features to new divisions, locations, or lines of business.
The defining fact: a system is ALREADY LIVE and the work increases the value it delivers.

## The firm's optimization methodology
A well-formed optimization opportunity fits the firm's published approach — a methodical assessment,
not an open-ended build:
1. **Assess issues** — identify the pain points and where the system underdelivers.
2. **Quantify gaps** — measure the distance between current state and leading practice.
3. **Prioritized improvement plan** — a sequenced roadmap of fixes, weighing process, people, and
   technology, not just configuration.
Common recommendations: upgrading existing software, adding new modules, adding best-of-breed
components, and extending existing features to new divisions/locations/lines of business.

## The boundary test (this is the skill's hardest job)
Optimization vs the neighbors — get this right or the route is wrong:
- **vs implementation:** is the system LIVE? Improving/extending a running system is optimization.
  A net-new build, a full replacement, or a re-implementation is `implementation`. "Extend to new
  divisions/locations/LOBs" is the cleanest optimization tell.
- **vs managed-services:** is this a BOUNDED improvement project with an end state, or ongoing
  operate/run/support? A defined optimization scope is this skill; a continuous support relationship
  is `managed-services`.
- **vs pre-implementation:** optimization acts on an existing system; pre-implementation is advisory
  before a build is committed.
If the document reads as "we have X live and want it to do more / do it better," it's optimization.

## Qualification rubric
- **System-is-live confirmation:** what platform is already in production, how long, which modules?
  No live system named = likely mis-routed (probably implementation or pre-implementation).
- **Optimization type:** upgrade | new modules | best-of-breed add | integration improvement |
  feature/footprint extension (new divisions/locations/LOBs) | adoption & process tuning.
- **Platform alliance fit:** Oracle Cloud, IFS Cloud, Deltek Costpoint, OneStream, Plex, Sage Intacct,
  Salesforce are the named alliances. Optimization on an alliance platform is strong fit; off-alliance
  is a partnering/capability flag.
- **Value & forcing function:** is there a measurable pain (data errors, downstream impact, manual
  inefficiency, unrealized ROI) or a trigger (acquisition to absorb, growth straining the system,
  a version going unsupported)? A quantified pain strengthens the case.
- **Disqualifiers / flags:** scope that's actually a net-new build mislabeled as "optimization";
  a system so broken the honest answer is re-implementation, not tuning; off-alliance with no path.

## ICP signals specific to optimization
General ICP (size, US-based, vertical) lives in `config/icp.py`. Strong optimization fit tends to
look like: an enterprise or upper-mid-market client already live on an alliance platform, hitting
limits from growth or acquisition, with a measurable gap between current use and leading practice —
especially extending a proven system to new parts of the business.

## Route to & package to compile
**Target team:** delivery / proposal team.
Compile into `team_package`:
- `live_platform`: the system already in production (+ version/age if stated)
- `optimization_type`: upgrade | modules | best-of-breed | integration | extension | adoption
- `pain_points`: the issues/gaps driving the work, quantified where possible
- `forcing_function`: acquisition / growth / unsupported version / unrealized ROI, or what's missing
- `alliance_flags`: off-alliance or partnering concerns
- `re_implementation_risk`: whether the "optimization" is really a rebuild in disguise
- `open_questions`: what's missing to scope

## Readout template
Lead with the verdict in one sentence a partner can act on in ten seconds (what's live, what kind of
optimization, what's driving it), then deal shape (assessment-led, phased improvement, likely size
and term — typically smaller and shorter than a full implementation), then the compiled package.
